No Oracle Database, esses indicadores aparecem em estatísticas de performance e ajudam a entender como o banco de dados acessa dados e processa transações. Eles são fundamentais para diagnosticar gargalos.

**db block gets**: representa leituras de blocos no modo atual, ou seja, o Oracle acessa o bloco mais recente na memória para fazer modificações. Esse tipo de leitura ocorre principalmente em operações com INSERT, UPDATE e DELETE. Valores altos indicam workload de escrita.

**consistent gets**: são leituras lógicas feitas para garantir consitência de leitura. O Oracle pode reconstruir versões anteriores dos dados usando undo para manter o isolamento transacional. Esse tipo de leitura é comum em SELECTs. Valores altos indicam workload de leitura.

**physical reads**: indicam quantas vezes o Oracle precisou ir ao disco buscar blocos que não estavam no buffer cache. Esse é um dos principais indicadores de I/O real. Valores altos geralmente indicam cache insuficiente ou queries ineficientes.

**redo entries**; representa o número de registros gerados no redo log buffer. Cada mudança no banco gera uma entrada de redo. Esse indicador mostra a intensidade de atividade transacional.

**redo size**: é o volume total de dados gerados no redo, medido em bytes. Enquanto redo entries conta eventos, redo size mede o peso dessas mudanças. Valores altos indicam grande volume de escrita ou transações pesadas.

No Oracle Database, redo é o registro sequencial de todas as mudanças feitas nos dados do banco. Ele representa um "log de reconstrução" que permite ao Oracle refazer operações em caso de falha.

Quando você executa um INSERT, UPDATE ou DELETE, o Oracle não grava apenas o resultado final nos datafiles. Ele primeiro registra a mudança no redo log buffer em memória. Depois, o processo LGWR grava esse conteúdo nos redo log files em disco. Esse fluxo garante a durabilidade das transações.

O ponto central é que o redo não armazena simplemente o dado final, mas sim informações suficientes para reproduzir a alteração. Se houver uma queda de energia ou falha da instância, o Oracle usa esses registros para aplicar novamente as mudanças que já tinham sido confirmadas, garantindo consistência.

É importante diferenciar redo de undo. O redo serve para refazer operações durante recovery. O undo serve para desfazer operações em caso de rollback ou para fornecer leitura consistente.

Na prática, o redo é o mecanismo que permite ao Oracle cumprir o princípio de durabilidade das transações. Sem ele, qualquer falha poderia resultar em perda de dados mesmo após um commit.

Após os seus teste com as querys disponível no conteúdo: **Missão - Deep Dive**, o fato de redo buffer allocation retries permanecer em zero é muito importante. 
Significa que o redo log buffer foi suficiente para suportar essa carga sem contenção. Em outras palavras, o processo LGWR conseguiu esvaziar o buffer rápido o bastante para não causar espera nas sessões.

A leitura correta é que seu workload gerou uma quantidade significativa de redo, como esperado em operações massivas de UPDATE, mas sem pressão sobre o redo buffer. Em um cenário com volume muito maior ou commits muito frequentes, esse último indicador poderia começar a subir, sinalizando gargalo de escrita.
