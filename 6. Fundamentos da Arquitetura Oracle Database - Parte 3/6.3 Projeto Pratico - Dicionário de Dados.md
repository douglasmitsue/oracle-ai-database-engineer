### Missão 7

O dicionário de dados no Oracle Database é um conjunto de tabelas e views internas que armazenam todos os metadados do banco. Em termos práticos, é "o banco que descreve o próprio banco".

Ele contém informações sobre estruturas e objetos como tabelas, índices, colunas, usuários, permissões, constraints, além de estatísticas e detalhes de execução.
Sempre que você cria, altera ou consulta um objeto, o Oracle lê ou atualiza esse dicionário.

Esse dicionário não é acessado diretamente pelas tabelas internas, mas por meio de views públicas organizadas em três níveis pricipais.

- USER_ mostra objetos do próprio usuário;
- ALL_ mostra objetos acessíveis ao usuário;
- DBA_ mostra tudo no banco, normalmente exigindo privilégios administrativos.

Além de metadados estruturais, o dicionário também expõe informações dinâmicas por meio das chamadas views de desempenho, com V$SESSION  e VSQL, que refletem o estado atual da instância.

Em termos operacionais, o dicionário de dados é essencial para o funcionamento do otimizador de consultas, controle de segurança, validação de objetos e administração geral do banco. Sem ele, o Oracle não conseguiria interpretar nem executar praticamente nenhuma operação SQL.

### Missão 9

A forma mais fácil de entender a relação entre CDB e PDB é pensar no multitenant como uma virtualização de bancos dentro de uma única instância do Oracle Database.

Antes do Oracle 12c, uma instância Oracle correspondia a um único banco. Com multitenant, uma única instância(memória, processos, background) pode hospedar vários bancos isolados.

Essa arquitetura é composta por três elementos principais:

1. O CDB(Container Database) é o banco "raiz". Ele não é usado para aplicação. Ele contém a infraestrutura global: controle de instância, memória, background processes, data dictionary global e metadados de todos os containers.

Dentro do CDB existem containers.

O principal é o CDB$ROOT. É onde você executa operações administrativas globais: 
abrir e fechar PDBs, criar PDBs, gerenciar storage, usuários comuns, etc. É exatamente por isso que você precisa estar nele para rodar o ALTER PLUGGABLE DATABASE.

2. Existe também o PDB$SEED, que é um template somente leitura. Sempre que você cria uma nova PDB, o Oracle copia esse seed.
3. E finalmente as PDBs (Pluggable Databases), como a FREEPDB1. Cada PDB é um ponto de vista lógico, um banco independente: tem usuários próprios, tablespaces próprios, objetos próprios e até ciclo de vida próprio (open, close).

A separação fundamental  é de escopo.

Quando você está no CDB$ROOT, você enxerga o ambiente como um todo. Views com 
v$pdbs mostram todas as PDBs. Comandos como:

ALTER PLUGGABLE DATABASE freepdb1 CLOSE;
só funcionam aqui.

Quando você está dentro de uma PDB, o contexto muda completamente. Você está "dentro de um banco isolado". Nesse momento:

- Você não consegue gerenciar outras PDBs
- Você não consegue executar comandos de nível global
- Você só enxerga objetos daquela PDB

Outro ponto importante é o tipo de usuário.

Usuários podem ser locais ou comuns.

- Usuários locais existem apenas dentro da PDB. Exemplo: <user>LAB
- Usuários comuns existem no CDB inteiro e começam com C##.

Quando você faz DROP USER, você está operando dentro da PDB, o que é correto, porque esses usuários pertencem à PDB.

Agora um modelo mental simples que resolve 90% dos problemas:

- CDB$ROOT é o "hypervisor"
- PDB é a "VM"
- Sessão é o "terminal dentro da VM"

Se você desliga a VM estando dentro dela, não funciona. Você precisa ir para o hypervisor.
