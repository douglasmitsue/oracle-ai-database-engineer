## 7. Missão 6 — Undo e Transações

> **Conceito central:** o Undo tem **dois papéis simultâneos**: desfazer transações (rollback) e oferecer leitura consistente a outras sessões.

Trabalharemos com **duas sessões em paralelo**. No DBeaver, abra **dois SQL Editors** usando a conexão `Oracle LAB`. Vamos chamá-los de **Sessão A** e **Sessão B**.

#### Importante: Se estiver utilizando o DBeaver como Client para as consultas na Base de Dados Oracle desabilitar o autocommit na sua conexão com o Oracle.

![[Pasted image 20260710120331.png]]


### 7.1. Preparação

Na **Sessão A**:

```sql
SELECT id, cliente, valor FROM vendas WHERE id = 1;
```

Anote o valor.

### 7.2. Rollback

Na **Sessão A**, comece uma transação mas **não** comite:

```sql
UPDATE vendas SET valor = 99999 WHERE id = 1;
-- NÃO DÊ COMMIT
SELECT id, valor FROM vendas WHERE id = 1;
```

A Sessão A enxerga `99999`. Agora, na **Sessão B**:

```sql
SELECT id, valor FROM vendas WHERE id = 1;
```

**Descoberta:** a Sessão B enxerga o **valor antigo**, não o `99999`. Como isso é possível, se o bloco no Buffer Cache já foi modificado? Resposta: o Oracle reconstruiu a versão antiga do bloco **usando o undo**.

Isso acontece por causa de isolamento de transações + MVCC (Multi-Version Concurrency Control) no Oracle Database.

Na Sessão A, você executou um UPDATE sem COMMIT. Esse novo valor (99999) existe apenas dentro do contexto da própria transação. Por definição, uma transação sempre “enxerga” suas próprias alterações, mesmo que ainda não tenham sido confirmadas.

Já a Sessão B está em outra transação independente. Para ela, o banco precisa garantir consistência de leitura. Isso significa que a query deve enxergar o estado dos dados como estava no momento em que começou, ignorando alterações não commitadas de outras sessões.

Embora o bloco no buffer cache já tenha sido modificado fisicamente, o Oracle mantém a versão anterior no Undo. Quando a Sessão B faz o SELECT, o banco usa esse Undo para reconstruir logicamente o valor antigo, como se o UPDATE da Sessão A ainda não existisse.

Agora, de volta na **Sessão A**:

```sql
ROLLBACK;
SELECT id, valor FROM vendas WHERE id = 1;
```

O valor voltou ao original e o Oracle **leu o undo e desfez a alteração**.


### 7.4. Inspecionando o Undo

Abra o SQLPlus:

```bash
docker exec -it oracle sqlplus system/<suasenha>pwd@//localhost:1521/FREEPDB1
```

Execute:

```sql
-- Tablespace de Undo em uso
SHOW PARAMETER undo_tablespace;

-- Retenção configurada
SHOW PARAMETER undo_retention;

```

No Oracle Database, o parâmetro undo_retention define por quanto tempo (em segundos) o Oracle tenta preservar os dados de Undo antes de reutilizá-los.

Agora acesse a conexão FREEPDB1:


```sql
-- Espaço e segmentos de undo
SELECT tablespace_name, status, ROUND(SUM(bytes)/1024/1024, 2) AS mb
FROM   dba_undo_extents
GROUP BY tablespace_name, status;

-- Transações em andamento
SELECT s.sid, s.username, t.start_time, t.used_ublk AS undo_blocks_usados
FROM   v$transaction t
JOIN   v$session s ON s.saddr = t.ses_addr;
```

## 8. Missão 7 — Dicionário de Dados

> **Conceito central:** o dicionário é o "sistema nervoso" do banco. Há duas naturezas distintas: views **estáticas** (`DBA_*`, `ALL_*`, `USER_*`) e views **dinâmicas de performance** (`V$`).

### 8.1. As três famílias estáticas (prefixo)

| Prefixo   | Mostra o quê                                                               |
|-----------|-----------------------------------------------------------------------------|
| `USER_*`  | Apenas os objetos do **usuário conectado**.                                 |
| `ALL_*`   | Objetos que o usuário conectado **pode enxergar** (próprios + concedidos).  |
| `DBA_*`   | **Tudo** no banco (requer privilégio administrativo).                       |

Conectado como `lab`:

```sql
SELECT table_name FROM user_tables;
SELECT owner, table_name FROM all_tables WHERE owner IN ('DSALAB','SYSTEM') FETCH FIRST 10 ROWS ONLY;

-- O próximo falha na sessão Oracle LAB pois DSALAB não tem privilégio DBA:
SELECT owner, table_name FROM dba_tables FETCH FIRST 10 ROWS ONLY;
```

Conectado como `system`:

```sql
SELECT owner, table_name FROM dba_tables WHERE owner = 'DSALAB';
```

### 8.2. Views dinâmicas de performance

Diferente das estáticas, as `V$` **não vêm do disco** — são uma janela para estruturas de memória da instância.

A primeira query abaixo lista as sessões de usuários conectadas ao Oracle naquele momento. Ela mostra identificadores da sessão (sid e serial#), usuário conectado, status da sessão, máquina de origem e programa usado para conexão. É útil para identificar quem está conectado, de onde veio a conexão e qual ferramenta ou aplicação está usando o banco.

A segunda query mostra os comandos SQL que mais consumiram I/O físico, ordenando pelo maior número de leituras em disco (disk_reads). Ela também exibe número de execuções, leituras lógicas em memória (buffer_gets) e o texto do SQL. É útil para encontrar queries potencialmente pesadas, que podem estar lendo muitos dados do disco e impactando performance.

A terceira query lista os principais eventos de espera acumulados no banco, ignorando eventos ociosos. Ela mostra a classe da espera, o evento específico, quantidade total de esperas e tempo total aguardado. É útil para entender onde o banco está gastando mais tempo esperando, por exemplo I/O, locks, rede ou concorrência interna.

```sql
-- Sessões ativas agora
SELECT sid, serial#, username, status, machine, program
FROM   v$session
WHERE  username IS NOT NULL;

-- SQL top consumidor de I/O
SELECT sql_id, executions, disk_reads, buffer_gets, sql_text
FROM   v$sql
ORDER BY disk_reads DESC
FETCH FIRST 5 ROWS ONLY;

-- Waits atuais
SELECT wait_class, event, total_waits, time_waited
FROM   v$system_event
WHERE  wait_class <> 'Idle'
ORDER BY time_waited DESC
FETCH FIRST 10 ROWS ONLY;
```

### 8.3. Relatórios de Performance

```sql
-- 1. Sessões ativas com evento de espera atual
SELECT
    s.sid,
    s.serial#,
    s.username,
    s.status,
    s.machine,
    s.program,
    s.sql_id,
    s.event,
    s.wait_class,
    s.seconds_in_wait
FROM v$session s
WHERE s.username IS NOT NULL
ORDER BY s.status, s.seconds_in_wait DESC;

-- 2. Top SQL por consumo de CPU
SELECT
    sql_id,
    executions,
    cpu_time / 1000000 AS cpu_seconds,
    elapsed_time / 1000000 AS elapsed_seconds,
    buffer_gets,
    disk_reads,
    SUBSTR(sql_text, 1, 100) AS sql_text
FROM v$sql
WHERE executions > 0
ORDER BY cpu_time DESC
FETCH FIRST 10 ROWS ONLY;



-- 3. Consumo de memória da instância Oracle
SELECT
    component,
    current_size / 1024 / 1024 AS current_size_mb,
    min_size / 1024 / 1024     AS min_size_mb,
    max_size / 1024 / 1024     AS max_size_mb,
    user_specified_size / 1024 / 1024 AS user_specified_size_mb
FROM v$memory_dynamic_components
WHERE current_size > 0
ORDER BY current_size DESC;

```

## 9. Missão 8 — Usuários, Schemas e Privilégios

> **Conceito central:** ao criar um **usuário**, o Oracle cria um **schema** de mesmo nome. Privilégios podem ser de sistema (o que você pode **fazer**) ou de objeto (em qual objeto você pode agir).

Use a conexão `Oracle FREEPDB1` como `system`.

### 9.1. Criando um usuário leitor

```sql
CREATE USER analista IDENTIFIED BY analista123
  DEFAULT TABLESPACE users
  QUOTA 0 ON users;

GRANT CREATE SESSION TO analista;
```

Tente conectar pelo DBeaver como `analista`. Ele entra, mas **não enxerga nada**.

### 9.2. Concedendo privilégios de objeto

Como `lab`:

```sql
GRANT SELECT ON dsalab.vendas TO analista;
```

Como `analista`:

```sql
SELECT COUNT(*) FROM dsalab.vendas;       
SELECT COUNT(*) FROM dsalab.vendas WHERE ROWNUM < 5;
-- Tentar atualizar falha:
UPDATE dsalab.vendas SET valor = 0 WHERE ROWNUM = 1;
```

### 9.3. Roles: privilégios agrupados

```sql
-- Como SYSTEM
CREATE ROLE leitor_vendas;
GRANT SELECT ON dsalab.vendas TO leitor_vendas;

-- Agora crie outro usuário e dê só a role
CREATE USER auditor IDENTIFIED BY auditor123;
GRANT CREATE SESSION TO auditor;
GRANT leitor_vendas TO auditor;

-- Teste select e update e depois revogue a role
REVOKE leitor_vendas FROM auditor;
```

### 9.4. Inspecionando o que cada usuário pode fazer

```sql
-- Privilégios de sistema
SELECT grantee, privilege
FROM   dba_sys_privs
WHERE  grantee IN ('DSALAB','ANALISTA','AUDITOR')
ORDER BY grantee, privilege;

-- Privilégios de objeto
SELECT grantee, owner, table_name, privilege
FROM   dba_tab_privs
WHERE  grantee IN ('DSALAB','ANALISTA','AUDITOR','LEITOR_VENDAS')
ORDER BY grantee;

-- Roles concedidas
SELECT grantee, granted_role
FROM   dba_role_privs
WHERE  grantee IN ('DSALAB','ANALISTA','AUDITOR');
```

## 10. Missão 9 — Startup e Shutdown

> **Conceito central:** um banco Oracle passa por três estágios ao subir — **NOMOUNT → MOUNT → OPEN** — e cada um habilita um conjunto diferente de operações. Em ambiente multitenant, cada **PDB** tem seu próprio ciclo de vida.

> **Atenção:** não vamos derrubar o CDB (isso encerraria o contêiner). Vamos operar o ciclo de vida da **PDB** `FREEPDB1`, que é rigorosamente o mesmo conceito, só num nível mais granular.

Use a conexão `Oracle CDB FREE` como `system`. Como alternativa você pode conectar como sysdba no SQLPlus:

docker exec -it oracle sqlplus sys/alunodsapwd@//localhost:1521/FREE as sysdba

SHOW CON_NAME;

### 10.1. Estado atual da PDB

```sql
SELECT name, open_mode, restricted, open_time
FROM   v$pdbs;
```

`FREEPDB1` deve estar em `READ WRITE`.

### 10.2. Fechando a PDB (equivalente a "shutdown")

```sql
ALTER PLUGGABLE DATABASE freepdb1 CLOSE IMMEDIATE;

SELECT name, open_mode FROM v$pdbs;
```

Agora `FREEPDB1` está em `MOUNTED`. **Teste abrir uma conexão DBeaver no `Oracle FREEPDB1` nesse momento** e vai falhar. Isso mostra concretamente o que significa uma PDB fechada.

### 10.3. Abrindo em READ ONLY

```sql
ALTER PLUGGABLE DATABASE freepdb1 OPEN READ ONLY;

SELECT name, open_mode FROM v$pdbs;
```

Reabra a conexão `Oracle Lab` e tente um `INSERT`:

```sql
INSERT INTO dsalab.vendas (cliente, produto, valor) VALUES ('X','Y',1);
-- ORA-16000: database or pluggable database open for read-only access
```

Leitura funciona, escrita não. É exatamente o comportamento de um **standby** aberto para relatórios.

### 10.4. Voltando para READ WRITE

```sql
-- De volta no CDB
ALTER PLUGGABLE DATABASE freepdb1 CLOSE IMMEDIATE;
ALTER PLUGGABLE DATABASE freepdb1 OPEN READ WRITE;

SELECT name, open_mode FROM v$pdbs;
```

### 10.5. Os três estágios vistos em outro nível (instância CDB)

Mesmo sem derrubar a instância, veja o log do alertlog para entender que, **quando o contêiner subiu**, o Oracle passou por NOMOUNT → MOUNT → OPEN:

```bash
docker exec -it oracle bash -c "tail -100 /opt/oracle/diag/rdbms/free/FREE/trace/alert_FREE.log"
```

Procure nas linhas por:

- `Starting ORACLE instance` → começo do **NOMOUNT**
- `ALTER DATABASE MOUNT` → transição para **MOUNT**
- `ALTER DATABASE OPEN` → transição para **OPEN**

## 11. Missão 10 - Desafio Final — Deep Dive Integrado

Agora que você viu cada peça isolada, o desafio é **amarrar tudo em uma só história**. Responda às perguntas rodando SQL no DBeaver — **o resultado deve ser uma linha única em cada item**, quando aplicável.

1. **Identidade:** Qual é o `DB_NAME`, a versão `V$INSTANCE.VERSION_FULL` e há quanto tempo (em minutos) essa instância está no ar?
2. **Do lógico ao físico:** Em qual arquivo físico mora a tabela `DSALAB.VENDAS`?
3. **Tamanho real:** Quantos blocos, quantas extensões e quantos MB a tabela `LAB.VENDAS` ocupa?
4. **Memória:** Qual componente da SGA é o maior neste exato momento?
5. **Processos:** Quantos background processes estão ativos agora?
6. **Cache:** Qual é o cache hit ratio **neste instante**?
7. **Transações:** Existe alguma transação não commitada agora? (Use `V$TRANSACTION`.)
8. **Undo:** Quantos MB a tablespace de Undo está usando em blocos `ACTIVE`?
9. **Segurança:** Quais privilégios de sistema o usuário `LAB` possui?
10. **Ciclo de vida:** Quais PDBs existem no CDB e qual é o `OPEN_MODE` de cada uma?

**Entregável sugerido:** um único arquivo `.sql` contendo as 10 queries, cada uma precedida por um comentário `-- Pergunta N:` — e um print do resultado.
